# 核心分工架构 (The Dream Team)

## 1. 模型角色分工

### DeepSeek V3.1
**角色**: 量化策略师 (Brain)
**核心必杀技**: 最懂 A 股黑话、数学逻辑最强。它对“打板”、“弱转强”、“分歧一致”的理解是原生的，不像 GPT 是翻译的。
**什么时候切换给它？**: 编写 backtest_engine 中的交易逻辑、止损算法、反操纵规则时。

### GPT-5.2 Codex
**角色**: 全栈工程师 (Hands)
**核心必杀技**: 工程架构稳、代码错误率低。它最擅长维护文件结构、写 Python 类封装、Docker 配置、API 接口。
**什么时候切换给它？**: 搭建项目框架、写爬虫基础代码、Debug 报错、重构文件目录时。

### Gemini 3 Pro
**角色**: 数据吞噬者 (Eyes)
**核心必杀技**: 超大上下文 (200k+)、多模态。它能一次性吃下整个 AkShare 文档，或者识别视频截图。
**什么时候切换给它？**: 阅读超长 API 文档、OCR 识别视频截图、分析几百行的报错日志时。

### Kimi k2-0905
**角色**: 舆情分析员 (Mouth)
**核心必杀技**: 超长中文文本理解、无幻觉摘要。它处理几万字的长篇研报或大量评论区内容时，总结最精准。
**什么时候切换给它？**: 清洗博主废话、从几千条评论中提取核心观点、分析长篇研报时。

## 2. Kimi k2-0905 在你项目中的具体用法
虽然 Kimi 写复杂 Python 代码可能不如 DeepSeek/GPT-5.2，但在你的**“舆情挖掘”**环节，它是神器。

### 场景 A：清洗博主的“废话文学”
博主可能发了 500 字，其中 400 字是情绪宣泄，只有 50 字是干货。
**指令**: 让 Kimi 提取摘要。
**Prompt**: "Kimi，这是博主的一篇长文动态。请剔除所有的情绪发泄和广告，只保留关于‘市场判断’和‘个股逻辑’的干货，不超过 50 字。"

### 场景 B：视频评论区风向标
如果你抓取了某热门视频下的 1000 条评论，想看散户情绪。
**指令**: 把 1000 条评论丢给 Kimi（它吃得消）。
**Prompt**: "分析这 1000 条评论，告诉我散户现在的情绪是‘恐慌’、‘贪婪’还是‘犹豫’？统计其中提到最多的 3 只股票。"

## 3. 实战演示：如何打这一套“组合拳”？
假设你要开发 Sprint C 中的 “贝叶斯信誉评分系统”，操作流程如下：

### 第 1 步：找 DeepSeek 设计算法 (逻辑层)
切换到 DeepSeek V3.1：
> "我是做 A 股量化的。请帮我设计一个基于贝叶斯更新的博主信誉评分公式。初始分是 Beta(2,2)。如果博主推荐的股票 T+1 日涨幅 > 2%，视为胜利；跌幅 < -2%，视为失败。请给出数学推导和 Python 实现函数，并解释为什么比简单的加减分好。"
(DeepSeek 会给你一套非常严谨的数学逻辑和算法代码)

### 第 2 步：找 GPT-5.2 落地代码 (工程层)
切换到 GPT-5.2 Codex：
> "这是 DeepSeek 设计的贝叶斯算法函数 [粘贴代码]。请把它集成到我的 core/strategy_analysis.py 文件中。要求：封装成 BloggerRater 类，增加异常处理，并写好单元测试。"
(GPT-5.2 会完美地把它放进你的项目结构里，处理好 import)

### 第 3 步：找 Kimi/Gemini 处理历史数据 (数据层)
切换到 Kimi 或 Gemini：
> "我现在有一份历史数据 strategy_log.csv (粘贴部分数据)。请帮我分析一下，如果用这个贝叶斯算法，哪些博主的分数会提升最快？给我一个模拟结果。"

## 4. 总结
*   **GPT-5.2 Codex**: 你的主操刀手，默认用它写代码。
*   **DeepSeek V3.1**: 你的军师，遇到不懂的 A 股逻辑、复杂的数学公式，问它，然后把答案给 GPT 去写。
*   **Gemini 3 Pro**: 你的工具人，用来读文档、看报错、做 OCR。
*   **Kimi k2**: 你的秘书，用来读长文、做舆情摘要。

**现在的建议**：既然你在 Trae 里，默认使用 GPT-5.2 Codex 进行文件编辑和代码编写（因为它对 IDE 的控制力最强）。当遇到策略逻辑卡壳或者需要理解 A 股特殊规则时，在对话框里临时切换模型到 DeepSeek 获得灵感，再切回 GPT 实现。
